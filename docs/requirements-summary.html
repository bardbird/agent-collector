业务要交付什么

交付物全景

核心目标是训练一个能使用工具完成真实任务的 Agent（非聊天机器人）， 因此数据需同时满足任务多样性与Agent 行为多样性。 全部 9 项均为 P0 优先级，按 SFT / RL 两条线交付。

2.1 通用硬指标红线（所有采购项适用）

  • 工具调用轮数不足
    每条轨迹 < 4 轮；平均 < 7 轮
    5.4 例外 2–6 轮
  • 纯 coding 无工具调用
    不包含 tool_calls 的纯代码轨迹
    禁止
  • 纯聊天无 workspace
    仅限 5.1 / 5.2 skills 类不允许
    禁止
  • 对话结尾错误
    最后一条消息不是 assistant role
    禁止
  • assistant 空消息
    content 为 null 且无 tool_calls
    禁止
  • 数据未脱敏
    含真实 API Key / Token / 密码明文
    禁止
  • 反思轨迹占比不足
    含"错误→回退→重试"模式占比 5–10%
    5.2 / 5.4 / 5.5 / 5.9

2.4 各采购项专属红线

5.1 Skills SFT 轨迹数据

  • user message 未通过 <system-reminder> 注入 skills 列表
  • 不同轨迹的 skills 列表完全相同（缺乏差异化）
  • 合格率 < 95%

5.2 Skills RL 轨迹数据

  • 缺 answer_gt / model_query 字段
  • 无沙盒环境描述
  • SKILL.md 缺失 / 脚本有逻辑错误 / 未提供可执行环境
  • verifier 不能 100% 跑通；反思轨迹不在 5–10% 区间

5.3 RL 高质量 QA 数据集

  • answer_gt 不唯一、不明确
  • verifier 输出非二值 0/1 判定
  • 缺沙盒环境描述

5.4 多模态多轮 Python 轨迹

  • 用户输入无图片（每条必须多模态）
  • Python 调用轮次非 2–6；问题不需要 Python 即可解决
  • 反思轨迹 < 5%
  • 操作类型占比失衡（裁剪/旋转/缩放、滤镜/色彩、合成/拼接、水印、透视变换、混合操作）

5.5 多模态 Python FC 强化

  • 缺完整的工具定义 + 工具执行模块
  • 缺验证脚本（如像素级对比 verifier_crop.py）
  • 操作占比失衡

5.6 多模态搜索 SFT

  • query 无搜索必要性（不搜也能回答）
  • 输出缺搜索意图识别 / 检索词改写 / 证据引用
  • 未带 mock 环境
  • 固定套路（同一搜索模式占比过高）

5.7 多模态搜索 RL

  • 多跳搜索问题 < 50%
  • 答案非客观、不可验证
  • 一次搜索即可获得的低难度问题
  • query 答案是网络上不存在的信息

5.8 工具调用泛化 SFT

  • 覆盖场景 < 10 类，或某场景占比 < 5%
  • 每个场景未提供 mock 的工具定义 / 输入参数 / 返回结果 / 标准轨迹
  • 未带 mock 环境

5.9 工具调用泛化 RL

  • 场景覆盖 < 1000
  • 缺 6 类特殊情形分布： 正常串联 50% 错误恢复 15% 冗余识别 10% 参数错误 10% 工具失效 10% 合适停止 5%

2.6 验收门槛（§8）

指标门槛抽检方式
Schema 合法率 100% 字段缺失/类型错误/tool_call_id 不匹配即不通过
内容合格率（人工抽检） ≥ 95% SFT 分层随机抽样，每批 3% 且 ≥ 100 条
Verifier 通过率（RL） ≥ 95% 100% verifier 自动校验 + 10% 人工复审
多样性达标 §4.4 全部满足 领域/工具/轮次/语言分布统计
反模式检测 未触发任何红线 自动扫描 4 大反模式
Skills 库抽检 20% skills，≥ 200 个 SKILL.md 完整性 + 脚本可执行性
!
退回与整改： 任一指标不达标 → 该批次整体退回，5 个工作日内提交整改版本； 同一批次 最多 2 次 整改机会，超过视为交付失败。
i
关键判断： 这份需求最严苛的"一票否决"点是 — (1) Schema 必须 100% 合法； (2) 四大反模式任一触发即整批退回； (3) 反思轨迹、工具轮数、领域分布等硬指标均有量化门槛； (4) RL 数据 verifier 必须跑通、Mock 覆盖率必须 100%。 一旦在多样性或可验证性上偷工减料，基本无法通过验收。

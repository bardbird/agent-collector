5000 个自研 Skills 的技术难点

~1500 多模态 · 80 子方向

1.1 分布约束的"硬数学" 高风险

文档要求 80 个二级子方向 × 每 ≥ 50 个 = 至少 4000 槽位，外加多模态 ~1500 跨领域。 这不是简单"凑数"，必须先做槽位规划表：

# 槽位规划示例（每子方向 ≥ 50 硬约束） D1 办公协作 750 → 假设 10 子方向 × 75 D2 信息研究 750 → 10 × 75 D3 数据处理 600 → 8 × 75 D4 内容创作 600 → 8 × 75 D5 代码工程 600 → 8 × 75 D6 文档理解 500 → 7 × ~72 D7 运维自动化 400 → 6 × ~67 D8 知识管理 300 → 5 × 60 ───── 4500 + 多模态 500 增量补足到 5000
!
难点：80 个子方向没有官方清单，需要供应商自行枚举并得到甲方认可， 否则后期验收"未覆盖某子方向"直接退回。

1.2 Skill 工程化的隐性成本 中风险

每个 Skill 不是写个 md 就行，交付标准是：

检查项实际工作量
SKILL.md 四段（名称/描述/Instructions/Examples/Limitations）平均 200–400 字
scripts/*.py 通过 py_compile至少 1 个能跑的脚本
requirements.txt + Python 版本依赖声明
关键逻辑人工审查抽检 20% ≥ 200 个
Docker 可运行依赖不能冲突
!
核心矛盾：5000 个 skill 的脚本依赖如果各自不同，容器化测试成本爆炸； 如果统一基础镜像，又会有 skill 因依赖缺失跑不起来。需要分层镜像策略。

1.3 去重判定的模糊地带（最大风险） 高风险

文档原文：功能相似度 > 80% 的 skills 视为重复，不计入有效数量

文档没有定义"相似度"的判定方式。可能是：

判定方式实现难度供应商风险
SKILL.md 文本余弦相似度（embedding）易最容易作弊，同领域天然聚集
参数 schema 相似度中中等，可通过参数命名绕过
行为相似度（输入输出对齐）难最难，但也最难作弊
!
风险：如果甲方用语义相似度（embedding），同领域 600 个 skill 天然容易聚集， 供应商可能交付 5000 个但实际有效只有 3500。 这条必须在 POC 阶段和甲方对齐判定算法。

1.4 多模态 Skill 的脚本化难题 中风险

~1500 个多模态 skill 要"分摊到 D1-D8"，但很多领域（如 D8 知识管理）天然多模态场景少。 强行分布会导致：

  • D8 里塞图像处理 skill → 与 D5/D6 重复
  • 跨领域相似度上升 → 触发去重红线
  • 视觉类 skill 的脚本依赖（PIL/OpenCV）需要在 Docker 镜像中预装

154 万条任务数据的技术难点

9 项 · SFT + RL

2.1 吞吐量级估算 中风险

# 数据量级估算 总量 154w 条 × 平均轮次 7 轮工具调用/条 × 每轮大小 1.5 KB（tool_calls + tool_response） ───────────────────────── ≈ 16 GB JSONL（不含图片） + 多模态资源 40-70w 张图（5.4 + 5.6） ≈ 总存储 50-100 GB
  • 单机处理不可行，必须分布式
  • JSONL 流式校验（schema 100% 合法率）
  • tool_call_id 全局唯一且配对，跨批次不能冲突

2.2 反模式红线是"批量生产的死敌" 高风险

四大反模式中，"模板化 query > 30%"和"单一操作范式 > 50%" 几乎是合成数据的天然倾向：

反模式LLM 合成倾向成因
模板化 query高同一 prompt 模板生成的 query 句式趋同
步骤处方式指令高LLM 喜欢"先做 A，再做 B"的处方式表述
单一操作范式高相似任务走相似工具链
跨领域行为雷同高领域换了但 agent 行为没换
!
结论：用单一 LLM 直接生成，几乎必然触发至少一条反模式红线。 必须用多策略种子 + 轨迹改写 + 主动去重。

2.3 反思轨迹不是"插入错误" 中风险

文档要求 5–10% 含"错误→识别→修正"。难点在于：

  • 错误必须是模型真实会犯的（参数错、工具选错、漏步骤）
  • 识别错误要有逻辑依据（看 tool response 推断）
  • 修正后路径要正确，不能"错误→瞎改→歪打正着"

要求反向构造：先有正确轨迹，再在中间环节注入"合理错误"，再让另一个 LLM 接续修正。 工作量是普通轨迹的 2–3 倍。

2.4 多模态资源生产 中风险

采购项图片需求难点
5.4 (20w)20w 张需多样化真实图片（照片/截图/插画）
5.6 (20-50w)20-50w 张含截图/表格/视频帧，版权风险
5.5 / 5.7复用上图输入图 + 验证输出图
!
版权红线：商用训练数据用网图有合规风险。 要么自研生成（AIGC 图片 + 后处理），要么采购授权数据集。

2.5 Mock 环境的规模爆炸 高风险

涉及 mock 的 4 项：5.6 (20-50w) + 5.7 (5w) + 5.8 (50w) + 5.9 (4w) = 80-110w 条轨迹。

# Mock 条目数估算 轨迹数 80w × 平均轮次 7 × 外部工具占比 0.4 ───────────────────── ≈ 224w 条 mock 响应
  • 一致性：同一 request_hash 必须返回同一 response（不能动态生成）
  • 覆盖率：100%，漏一条就退回整批
  • 真实性：mock 内容要像真实 API 返回，否则模型学不到东西

2.6 Verifier 工程化 中风险

5 类 verifier 覆盖 ~14w 条 RL 数据，每类逻辑不同：

类型适用难点
exact_match数值/日期格式要对齐（"640kW" vs "640 kW"）
model_judge开放回答需要评判模型 + prompt，评判模型本身的偏差
script图像/文件每个场景写专门脚本（如 verifier_crop.py）
!
model_judge 是最大坑：评判模型若用 GLM-4，它对"YES/NO"的回答有位置偏差， 需要做位置互换校准（同一样本问两遍，互换 YES/NO 位置取一致结果）。

模型数据采集的核心难点

真实 vs 合成

3.1 "合成"与"真实"的鸿沟 高风险

需求反复强调"真实环境中使用工具完成任务的 Agent"。但批量数据只能是合成的。鸿沟在于：

真实轨迹
  • 用户 query 多样、模糊、有错别字
  • agent 会探索、会卡住
  • 工具会失败、会超时
  • 路径有冗余、有回退
合成轨迹（默认倾向）
  • query 工整、表述清晰
  • agent 路径最优、无探索
  • 工具调用必成功
  • 无无效尝试

对策：引入"噪声注入"——给 query 加口语化/错别字/多轮澄清， 给轨迹加"无效尝试→放弃→换路径"。

3.2 自我一致性陷阱 高风险

单个 LLM 生成 154w 条，会出现：

  • 同一领域内 query 高度相似（embedding 聚簇）
  • 工具调用顺序趋同（LLM 的"偏好路径"）
  • 反思模式套路化（"我发现参数错误，于是修正"）

必须多模型协同：seed 用 A 模型、轨迹展开用 B 模型、反思用 C 模型、校验用 D 模型， 避免单一偏好污染。

3.4 多模态对齐 中风险

图像理解轨迹的难点：模型生成的 tool_call 参数（如 bbox 坐标）必须与图片实际内容对齐。 纯 LLM 文本生成无法保证，需要：

  • 视觉模型先理解图片 → 生成 ground truth 坐标
  • 再构造轨迹让"模型"输出接近 ground truth 的参数
  • 验证脚本做像素级对比（如 verifier_crop.py）

批量规划方案（可执行）

流水线 · 门禁 · 成本

4.1 Skill 库生产流水线（5000 个）

Phase S1 · 槽位规划（前置，必须先和甲方对齐）

  • 枚举 80 个二级子方向清单
  • 多模态 1500 个的跨领域分布表
  • 去重相似度判定算法（关键，POC 阶段锁定）

Phase S2 · 模板化生产

  • 每个领域设计 3–5 个可执行 skill 模板
  • 用 LLM 按槽位生成 SKILL.md 初稿
  • 脚本生成：通用工具（HTTP/文件/数据处理）复用库 + 领域特定逻辑

Phase S3 · 去重与质量

  • 自动：embedding 相似度 + 参数 schema 相似度双指标过滤
  • 人工：20% 抽检（≥ 200 个）SKILL.md 完整性 + 脚本可执行

Phase S4 · 容器化测试

  • 分层镜像策略：基础镜像（Python 3.10）+ 领域镜像（CV/NLP/数据）+ skill 专属层
  • 每 skill 独立 venv 安装 requirements.txt
  • py_compile + 冒烟测试（1–2 个 example 跑通）

4.3 Mock 环境批量生产

  • 预生成策略：先跑轨迹收集所有 (tool_name, arguments) 对，再批量生成 response
  • 真实 API 采样：合成阶段用真实 Google Lens / web_search 采样（需求允许），固化为 mock
  • 一致性 hash：sha256(tool_name + json.dumps(args, sort_keys=True))，去重存储
  • 覆盖率检查：check_mock_coverage.py 遍历 JSONL 所有 tool_calls，100% 命中才交付

4.4 Verifier 批量集成

  • 注册中心：每条 RL 数据的 meta.verifier_type 指向已注册 verifier
  • 回归测试集：每类 verifier 维护 100 条人工标注 gold set，每次跑流水线先过 gold set
  • model_judge 校准：评判模型做位置偏差校准（互换 YES/NO 位置取一致结果）
